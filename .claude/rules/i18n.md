# Internationalization (i18n) Rules

## Grundprinzip
- Mehrsprachigkeit von Anfang an einplanen — nachträglich ist teuer
- Alle user-facing Strings in Translation-Dateien, nie hardcoded
- Sprache ist User-Preference, nicht URL-basiert (es sei denn SEO relevant)

## Tech-Entscheidung (ADR erforderlich)
Vor Implementierung entscheiden und dokumentieren (in `docs/adr/`):

| Library | Am besten für | Vorteil |
|---------|--------------|---------|
| **next-intl** | Next.js (App Router) | Beste RSC-Integration, Server Components native |
| **i18next + react-i18next** | React SPAs, Remix | Größtes Ökosystem, viele Plugins |
| **Custom Translator** | Einfache Apps, Games, Vanilla JS | Minimaler Footprint (< 2KB), volle Kontrolle |
| **Paraglide.js** | Vite, SvelteKit | Compiler-basiert, tree-shakes unused translations |

**Empfehlung:** Für Next.js → `next-intl`. Für Games/Canvas-Apps → Custom Translator (weniger Overhead). Für React SPAs → `i18next`.

## Translation-Datei-Struktur

Einfach (für kleine Projekte / Games):
```
src/
  i18n/                    # oder src/messages/
    de.json                # Deutsche Übersetzungen (Fallback)
    en.json                # Englische Übersetzungen
    [locale].json          # Weitere Sprachen
```

Namespace-basiert (für große Projekte):
```
src/
  messages/                # oder src/i18n/
    de/
      common.json          # Shared (Navigation, Buttons, Errors)
      auth.json            # Login, Register, Password Reset
      [feature].json       # Feature-spezifische Texte
    en/
      common.json
      auth.json
      [feature].json
```

**Wähle die Struktur die zum Projekt passt** — Ordnername und -ort sind flexibel, Konsistenz ist wichtiger als Convention.

## Regeln
- **Keys:** dot-notation, englisch, beschreibend: `auth.login.button`, `game.action.walk.started`
- **Keine Konkatenation:** Nie `t('hello') + name` — immer `t('hello', { name })`
- **Pluralisierung:** ICU MessageFormat nutzen: `{count, plural, one {# item} other {# items}}`
- **Datums-/Zahlenformatierung:** `Intl.DateTimeFormat` / `Intl.NumberFormat` — nie manuell formatieren
- **Relative Zeit:** `Intl.RelativeTimeFormat` für "vor 5 Minuten", "in 3 Stunden"
- **Währung:** `Intl.NumberFormat` mit `style: 'currency'` — nie Symbol hardcoden
- **RTL-Support:** Nur wenn explizit gefordert — `dir="auto"` auf `<html>`
- **Fallback:** Default-Locale definieren (z.B. `de`), fehlende Keys zeigen Fallback-Sprache

## Server vs. Client
- **Server-rendered Apps (Next.js, Remix):** Locale aus Cookie/Header lesen, Translations server-side laden
- **Client-only Apps (Vite, SPAs, Games):** Translations beim App-Start laden, minimalen Bundle pro Locale
- **API Responses:** Fehler-Messages als Codes (`error.auth.invalid_credentials`), Client übersetzt
- **DB-Content:** User-generated Content wird NICHT übersetzt — nur UI-Strings

## Testing
- [ ] Alle Locales rendern ohne Fehler
- [ ] Fehlende Keys: Lint-Regel oder CI-Check (alle Keys in allen Locales vorhanden)
- [ ] Lange Texte: Deutsche Texte sind ~30% länger als englische — Layout testen
- [ ] Sonderzeichen: Umlaute (ä, ö, ü, ß), Akzente (é, à), CJK (wenn geplant)
- [ ] Zahlen/Datum: Locale-korrekte Formatierung (1.000,00 vs 1,000.00)
