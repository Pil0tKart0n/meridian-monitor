# Accessibility Rules

## WCAG Compliance (HARD RULE)
- **Minimum WCAG 2.2 Level AA** für alle user-facing Features
- Keine Ausnahme ohne AcDR (Accessibility Decision Record) in `docs/adr/`
- Rechtliche Anforderungen prüfen: BFSG/EAA (ab Juni 2025 in DE/EU), ADA, Section 508
- A11y ist kein Feature — es ist ein Qualitätsattribut jedes Features

## Color & Contrast (HARD RULE)
- **Text (normal):** Kontrast ≥ 4.5:1 gegen Hintergrund
- **Text (large, ≥18pt / 14pt bold):** Kontrast ≥ 3:1
- **UI-Komponenten (Icons, Borders, Focus):** Kontrast ≥ 3:1
- **Nie Farbe als einziges Unterscheidungsmerkmal** — immer zusätzlich Icon, Pattern oder Text
- Dark Mode: separat geprüft (andere Palette, andere Kontrast-Werte)
- Color-Vision-Deficiency (CVD): Deuteranopia + Protanopia simulieren
- Kontrast-Matrix in `docs/a11y/contrast-matrix.md` bei Projekt mit Design-System

## Keyboard Navigation (HARD RULE)
- **Alle interaktiven Elemente** per Keyboard erreichbar (Tab, Enter, Space, Escape, Arrow)
- **Kein Keyboard-Trap** — User kann jedes Element verlassen (Escape für Modals)
- **Focus-Indicator immer sichtbar:** min 2px Outline, Kontrast ≥ 3:1 gegen angrenzende Farben
- **Logische Tab-Reihenfolge:** folgt visueller Layout-Reihenfolge
- **Skip Links:** "Skip to main content" als erstes fokussierbares Element
- **`tabindex` Regeln:** nur `0` (in Tab-Order) oder `-1` (programmatisch). Niemals positive Werte.
- **`:focus-visible`** statt `:focus` (Focus nur bei Keyboard sichtbar, nicht bei Maus-Click)

## Semantic HTML (HARD RULE)
- **HTML-Semantik zuerst, ARIA nur als Ergänzung** (Erste ARIA-Regel: benutze kein ARIA wenn HTML reicht)
- `<button>` statt `<div onClick>`, `<a href>` statt `<span onClick>`
- `<nav>`, `<main>`, `<header>`, `<footer>`, `<aside>` für Landmarks
- `<h1>`–`<h6>` in korrekter Hierarchie (keine Ebenen überspringen)
- `<label>` mit `htmlFor` für jedes Formularfeld
- `<fieldset>` + `<legend>` für Gruppen von Formularfeldern
- `<html lang="de">` (oder entsprechende Sprache) gesetzt

## ARIA Patterns (HARD RULE)
- **aria-label / aria-labelledby** für Elemente ohne sichtbares Label
- **aria-describedby** für zusätzliche Beschreibungen (Fehlermeldungen, Hinweise)
- **aria-live="polite"** für dynamische Updates, `"assertive"` nur für kritische Fehler
- **aria-hidden="true"** nur für dekorative Elemente (Icons neben Text, Ornamente)
- **aria-expanded / aria-selected / aria-checked** für interaktive Zustände
- Keine falsche Rolle: `role="button"` nur wenn `<button>` nicht möglich ist

## Forms & Errors (HARD RULE)
- **Jedes Input hat ein sichtbares `<label>`** — kein Placeholder als Ersatz
- **Pflichtfelder:** `aria-required="true"` + visueller Indikator (nicht nur Sternchen)
- **Fehlermeldungen:** `aria-describedby` verknüpft + `aria-live` Region
- **Error Summary:** Bei Submit-Failure → Focus auf Fehlerzusammenfassung
- **Autocomplete:** `autocomplete` Attribute für Standard-Felder (name, email, tel)
- **Inline-Validation:** Fehler als `aria-live="polite"` announced, nicht nur Farbänderung

## Motion & Reduced Motion (HARD RULE)
- **`prefers-reduced-motion` MUSS respektiert werden** — Alternativ-Darstellung, nicht nur deaktiviert
- **Kein Flackern > 3x/Sekunde** (WCAG 2.3.1 — Epilepsie-Risiko)
- **Auto-playing Content:** Pause/Stop-Mechanismus bereitstellen
- **Parallax, Zoom, Rotation:** Nur mit reduced-motion Alternative

## Images & Media (HARD RULE)
- **Alle `<img>` haben `alt` Attribut** — beschreibend für informative, `alt=""` für dekorative
- **Alt-Text:** Beschreibt den Inhalt, nicht das Format ("Balkendiagramm zeigt monatlichen Umsatz" nicht "chart.png")
- **Video:** Untertitel (Captions) für Gehörlose
- **Audio:** Transkript bereitstellen

## Touch & Mobile (HARD RULE)
- **Touch Targets:** min 44x44px (WCAG), 8px Abstand zwischen Targets
- **Pinch-to-Zoom nicht deaktivieren:** Kein `maximum-scale=1.0` im Viewport
- **Komplexe Gesten:** Alternative bereitstellen (Swipe → auch Button möglich)
- **Orientierung:** Nicht sperren (außer essentiell, z.B. Kamera-App)

## Testing (HARD RULE)
- **Automatisiert:** axe-core in Playwright, Lighthouse A11y ≥ 90, in CI
- **Keyboard:** Jeder User-Flow komplett mit Keyboard-only navigierbar
- **Screen Reader:** Mindestens 1 Screen Reader getestet (NVDA Windows / VoiceOver macOS)
- **Zoom:** 200% Zoom ohne Inhaltsverlust oder horizontales Scrollen
- **Playwright MCP:** `browser_snapshot` für Accessibility-Tree-Checks nach UI-Änderungen

## Anti-Patterns (vermeiden)
- **A11y als Afterthought:** "Machen wir nach dem Launch" → in Phase 2 starten, nicht Phase 5
- **Nur axe-core:** Automatisierte Tools finden nur ~35% der Issues → Keyboard + SR Tests Pflicht
- **Positive tabindex:** `tabindex="3"` → erzeugt unvorhersehbare Tab-Reihenfolge
- **ARIA auf allem:** `aria-label` auf jedem `<div>` → Semantic HTML zuerst
- **Disabled ohne Erklärung:** Disabled Button ohne Kontext warum → Tooltip oder Text
- **Kontrast nur Light Mode:** Dark Mode hat andere Palette → separat prüfen
- **"Screenreader-User nutzen das eh nicht":** 15% der Weltbevölkerung hat eine Behinderung
