# Internationalization (i18n)

## Overview

WW3News targets a German-speaking primary audience with international reach. This epic sets up full internationalization with German (default) and English as launch languages, using next-intl for the Next.js App Router. All user-facing strings are extracted into translation files, dates and numbers are locale-formatted, and SEO is locale-aware with hreflang tags.

The architecture supports adding more languages later without code changes — only new translation files are needed.

## User Stories

### I18N-1: next-intl Setup with DE + EN

**As a** developer, **I want to** set up next-intl with German and English locales, **so that** all UI strings are translatable.

**Acceptance Criteria:**

- [ ] next-intl installed and configured for Next.js App Router
- [ ] Default locale: `de` (German)
- [ ] Supported locales: `de`, `en`
- [ ] Locale detection: browser `Accept-Language` header for first visit, cookie for returning users
- [ ] URL strategy: prefix-based (`/de/...`, `/en/...`) with `de` as default (optional: hide `/de/` prefix)
- [ ] Middleware configured for locale routing (`src/middleware.ts`)
- [ ] Translation files structure:
  ```
  src/messages/
    de/
      common.json      # Navigation, buttons, shared UI
      landing.json      # Landing page copy
      feed.json         # News feed UI
      auth.json         # Login, register, profile
      map.json          # Conflict map UI
      dashboard.json    # Economic dashboard UI
      errors.json       # Error messages
    en/
      common.json
      landing.json
      feed.json
      auth.json
      map.json
      dashboard.json
      errors.json
  ```
- [ ] `useTranslations()` hook available in Client Components
- [ ] `getTranslations()` available in Server Components
- [ ] Fallback: missing `en` key falls back to `de` value (not empty string)

### I18N-2: Language Switcher Component

**As a** user, **I want to** switch between German and English, **so that** I can use the platform in my preferred language.

**Acceptance Criteria:**

- [ ] Language switcher in the main navigation (header)
- [ ] Displays current language as flag icon + language code (DE / EN) or full name
- [ ] Dropdown or toggle between two languages
- [ ] Switching language: preserves current page and query params, only changes locale prefix
- [ ] Selected language persisted in cookie (`NEXT_LOCALE`)
- [ ] Responsive: compact on mobile (flag + code only), expanded on desktop
- [ ] Accessible: `<select>` or button group with `aria-label="Sprache wechseln"` / `"Switch language"`
- [ ] No full page reload on switch (client-side navigation)

---

### I18N-3: All UI Strings Extracted to Translation Files

**As a** developer, **I want to** have all user-facing strings in translation files, **so that** no hardcoded strings exist in components.

**Acceptance Criteria:**

- [ ] Zero hardcoded user-facing strings in React components
- [ ] Translation key naming convention: `[feature].[component].[element]`
  - Examples: `feed.filter.categoryAll`, `auth.login.submitButton`, `landing.hero.headline`
- [ ] Parametric strings use ICU format: `t('feed.articles.count', { count })` renders "1 Artikel" / "5 Artikel"
- [ ] Pluralization via ICU MessageFormat: `{count, plural, one {# Artikel} other {# Artikel}}`
- [ ] No string concatenation for translatable text
- [ ] Lint rule or CI check: detect hardcoded strings in JSX (eslint-plugin-i18next or equivalent)
- [ ] All error messages from `errors.json`, not inline
- [ ] Button labels, headings, tooltips, aria-labels — all from translation files

---

### I18N-4: Date/Number Formatting per Locale

**As a** user, **I want to** see dates and numbers formatted according to my locale, **so that** the platform feels native to my language.

**Acceptance Criteria:**

- [ ] Dates formatted via `Intl.DateTimeFormat` or next-intl's `useFormatter()`
  - DE: `28. Marz 2026`, `28.03.2026`
  - EN: `March 28, 2026`, `03/28/2026`
- [ ] Relative dates via `Intl.RelativeTimeFormat`: "vor 5 Minuten" / "5 minutes ago"
- [ ] Numbers formatted via `Intl.NumberFormat`
  - DE: `1.234,56`
  - EN: `1,234.56`
- [ ] Currency: EUR as default, formatted per locale
  - DE: `9,99 EUR`
  - EN: `EUR 9.99`
- [ ] Percentage: `62,4 %` (DE) vs `62.4%` (EN)
- [ ] All formatting centralized in utility functions or next-intl formatters (not manual formatting)
- [ ] GEI values, market prices, article dates — all locale-formatted

---

### I18N-5: SEO — hreflang Tags and Locale-aware Meta Tags

**As a** search engine crawler, **I want to** see correct hreflang annotations, **so that** users are served the right language version.

**Acceptance Criteria:**

- [ ] Every page has `<link rel="alternate" hreflang="de" href="...">` and `hreflang="en"`
- [ ] `<link rel="alternate" hreflang="x-default">` points to the German version (default)
- [ ] `<html lang="de">` or `<html lang="en">` set correctly per locale
- [ ] `<title>` and `<meta name="description">` translated per locale
- [ ] Open Graph tags (`og:title`, `og:description`) translated per locale
- [ ] `og:locale` set to `de_DE` or `en_US`
- [ ] Canonical URLs include locale prefix
- [ ] `sitemap.xml` includes all locale variants of each page
- [ ] Implemented via next-intl's metadata integration and `generateMetadata()`
- [ ] Verified: Google Search Console shows no hreflang errors (post-deployment check)

## Technical Notes

- **next-intl version:** Use latest stable version compatible with Next.js 15. Follow next-intl's App Router setup guide.
- **Message loading:** Server Components load messages at the layout level (`getMessages()`). Client Components receive via `NextIntlClientProvider`.
- **Type safety:** Generate TypeScript types from translation keys using `next-intl/plugin` or a custom script. Prevents typos in translation keys.
- **Translation workflow:** For MVP, translations are maintained manually in JSON files. For scale, consider Crowdin or Lokalise integration.
- **Text expansion:** German text is ~30% longer than English. Test all layouts with German text to ensure nothing overflows.
- **RTL:** Not planned for launch. Architecture supports adding RTL locales later (Arabic, Hebrew) with `dir="rtl"` on `<html>`.
- **Content translation:** Article content (from RSS/GDELT) is NOT translated — only UI strings. Articles are displayed in their original language. Future: GPT-powered article translation as a Premium feature.
- **ADR required:** URL strategy (prefix-based vs subdomain vs cookie-only), translation management tooling for post-MVP
